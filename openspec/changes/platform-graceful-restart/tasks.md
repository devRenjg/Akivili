- [ ] 1.1b 安全在线备份：改用 SQLite online backup API 或 `VACUUM INTO`（**不可直接 cp 主库**——WAL 下会漏未 checkpoint 的 `-wal` 数据）；建表/加列前先备份
- [ ] 1.1c 加索引：`run_logs(run_id,id)`、`run_queue(status,next_retry_at,id)`、active 唯一索引**两组互补**（第六轮 P1-1 conversation 粒度）——`UNIQUE(conversation_id, agent_slug) WHERE conversation_id IS NOT NULL AND status IN ('queued','claimed','running')` + `UNIQUE(task_id, agent_slug) WHERE conversation_id IS NULL AND status IN ('queued','claimed','running')`（先补 `run_queue.conversation_id` 列并归并存量，见 1.5b②）
- [ ] 1.2 **原子 claim(CAS)**：`_claim_one` 改单语句条件更新（`UPDATE...WHERE id=(子查询) AND status='queued' RETURNING *`），杜绝双领
- [ ] 1.3 **统一入队 + 两段式 dispatch**：所有触发（人工@/auto/mention/leader）经队列;`POST /tasks/{id}/dispatch` 接 idempotency key、幂等持久化用户消息、入队、返回 `execution_id`,**不在请求内跑 CLI**;`GET /executions/{execution_id}/events` 独立 SSE 订阅
- [ ] 1.4 **触发合并（不丢，非丢弃；含 claimed 与完整终态折叠，Review P1-4 + 第七轮 P0-3）**：同 (conversation,agent)（第六轮 P1-1 粒度，NULL 走 task 兜底）已有 queued→原子 `MAX/COALESCE` 合并消息水位到该行;**claimed/preparing→同样记 pending intent**（不能漏在 queued/running 两分法之外）;running→持久化 `rerun_requested`+`pending_through_message_id`+触发来源;收尾事务内据 pending 至多建一个 successor。各终态折叠规则：**failed**→仍据 pending 建 successor（不丢执行期间外部新意图）;**killed**→取消 pending 不建（用户主动停是唯一取消路径）;**superseded**→pending 由 recovery child 继承不另建;**poisoned failure**→自动恢复 successor 不建，但外部 pending intent SHALL 清 session 后建一个 `full_replay` 普通 successor 承接（第七轮 P0-3：poisoned 可禁自动重试但不能丢用户新指令）。**单 pending intent 的安全前提 = task:conversation 一对一**（见 1.4b）
- [ ] 1.4b **🔴 固化 task:conversation 一对一 DB 约束（Review 第七轮 P0-3，用户拍板）**：现有 `create_task`/`create_subtask` 每 task/subtask 新建独立 conversation（产品事实已 1:1），但 `tasks.conversation_id` 可空无唯一约束；不固化则同 conversation 多 task 会让单 pending intent 压掉/错投跨 task 触发（丢 task_id/prompt/idempotency/优先级/顺序）。迁移 SHALL 先清历史重复再建 `CREATE UNIQUE INDEX ... ON tasks(conversation_id) WHERE conversation_id IS NOT NULL`，使 conversation 粒度与 task 粒度一一对应、单 pending intent 才安全。**不改共享模型**（若未来确需共享再走 active 只限 claimed/running+多 queued 排队 或 FIFO `pending_intents` 表）;探针 `task_conversation_uniqueness_probe`(两 task 复用同 conversation_id 被 DB 约束拒绝、迁移无重复)、`cross_task_conversation_intent_probe`(1:1 下 B 属独立 conversation/独立 active、不被折叠错投给 A、不丢)、`poisoned_pending_intent_probe`(poisoned 时自动恢复停但外部 pending 生成 full_replay 普通 successor)、`failed_pending_not_lost_probe`(failed 时 pending 仍建 successor 不丢)
- [ ] 1.4a **状态消费者矩阵 + 完整转换表前移到阶段 1（Review P0-A/P1-5 + 第七轮 P1-5）**：本阶段引入 claimed/superseded/recovery_blocked 的同时，同步更新所有状态消费者（progress 聚合、Runtime 总览、任务自动流转、孤儿巡检、失败归因、前端 RunRow.vue/Runtime.vue 状态色）识别新状态 + 落地**完整 source-status 转换表**（每条转换的 source status + 目标 status + owner/generation/instance/lease 条件）。**SHALL NOT 把终态转换一刀切为 `WHERE status='running'`**——`claimed→queued`(claim lease 回收)/`claimed→killed`(claimed 被 kill)/`claimed→failed`(prestart 终局) 的 source 是 `claimed` 非 `running`;`running/claimed→queued`(瞬时 retry 回队) 两种 source 都合法。**不留到阶段 6**（阶段 6 的同名任务删除，见 6.x）;探针 `transition_source_status_probe`（claimed kill/prestart fail/claim lease reclaim 各自只接受合法 source status + owner/generation 条件，不因写死 running 漏改误改）
