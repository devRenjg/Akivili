# agent-self-improvement-metrics (delta)

## ADDED Requirements

### Requirement: 单 Agent 效率成长快照（L1）

系统 SHALL 能以单个 Agent 为主语，基于现有痕迹（`task_runs`/`activities`/`messages`）零埋点聚合出「同类任务效率趋势」，输出只读诊断快照。快照 SHALL 至少包含：同类任务耗时趋势、返工次数、求助次数、对话轮次。快照 SHALL 标注样本量，样本不足时只做「最近 vs 最早」粗对比而非拟合曲线。

#### Scenario: 生成单 Agent 体检快照
- **WHEN** 请求某 Agent 在某任务类型下的成长快照
- **THEN** 返回该 Agent 在该类型下的耗时/返工/求助/轮次的早期与近期对比，并标注样本量与存疑项

#### Scenario: 效率指标须联合解读防成本转移
- **WHEN** 呈现「耗时下降」
- **THEN** 同时呈现返工与求助趋势，使「又快又不返工又少求助」才被视为真变强，单项改善标注为存疑

### Requirement: knowhow 复用率作为核心信号（L1）

系统 SHALL 计算单个 Agent 的 knowhow 复用率 = 被后续任务显性援引过的 knowhow 条数 / 已沉淀 knowhow 总条数，并 SHALL 一并给出复用分布与「僵尸经验」（沉淀但从未被援引）清单。复用识别 SHALL 采用引用级（后续任务产出中显性援引），并 SHALL 明确标注该指标为「显性援引」而非「因果贡献」。

#### Scenario: 计算复用率与僵尸经验
- **WHEN** 请求某 Agent 的 knowhow 复用情况
- **THEN** 返回复用率、复用分布、以及从未被后续任务援引的 knowhow 清单

#### Scenario: 标注识别强度局限
- **WHEN** 呈现 knowhow 复用率
- **THEN** 附带说明「本指标为显性援引，负迁移在此层不可见，需反事实探针（L3）验证因果」

### Requirement: 度量不得接入 Agent 奖励（防 Gaming）

度量层产出的所有指标 SHALL 仅用于人工观测与诊断，MUST NOT 作为 Agent 的直接优化目标或奖励信号，以防指标被 Gaming（Goodhart）。

#### Scenario: 指标用途边界
- **WHEN** 任何模块试图消费度量层输出
- **THEN** 仅允许用于展示/人工诊断，不得回灌为 Agent 的执行奖励或自动策略更新的直接目标

### Requirement: 任务分型作为可比性前置（L2）

系统 SHALL 提供任务分型能力，使「同类任务」纵向对比有意义。分型 SHALL 采用**两级可延展结构**：阶段A（起步）以 **assignee 角色 slug 作为类型标签**（真实数据显示每个 Agent 当前任务性质单一，角色粒度恰好等于任务类型，零成本）；阶段B（当某角色任务变杂时）在该角色内按标题关键词或执行痕迹再分子类型。分型不得一次定死，须保留细分演进口。L1 快照 SHALL 依赖分型；无分型时应显式声明对比口径不可靠。

> 背景：纯标题关键词分型在真实数据上会撞车（多任务都含「直播日历」却分属取数/前端/协调），故起步不用关键词、改用角色 slug。

#### Scenario: 按类型分桶对比
- **WHEN** 生成某 Agent 的效率趋势
- **THEN** 仅在同一任务类型（起步=角色 slug）内做纵向对比，跨类型不直接比较耗时

#### Scenario: 角色内任务变杂时细分
- **WHEN** 某角色的任务开始跨越明显不同的性质（如 PO 既做协调又做数据）
- **THEN** 在该角色类型下按标题关键词/执行痕迹再分子类型，而非把异质任务混为一类

### Requirement: 仪式/非工作任务过滤（L2 前置）

成长度量 SHALL 先过滤掉仪式/非工作任务（如「自我介绍」「Kickoff」），只对真实工作任务计量。过滤采用**启发式 + 执行痕迹兜底**：标题含「自我介绍/Kickoff」判为仪式（强信号）；`description` 为空**且无任何 `task_runs`**（从未真跑过执行）方判为疑似仪式；空描述但有 run 记录的仍属真实工作任务，不得误杀。

> 背景：真实数据里 task 64「更新直播日历数据」desc=0 却是真实工作（有 run），故 desc=0 不能作为硬过滤条件，须叠加「无 run」兜底。

#### Scenario: 过滤仪式任务
- **WHEN** 聚合某 Agent 的成长数据
- **THEN** 标题为「自我介绍/Kickoff」的任务被排除，不计入效率与复用率统计

#### Scenario: 空描述但真干过活的任务不被误杀
- **WHEN** 某任务 `description` 为空但存在 `task_runs`（真执行过）
- **THEN** 该任务仍被计入成长度量，不因空描述被判为仪式任务

### Requirement: 度量单点先行（L2 现实约束）

系统 SHALL 允许成长度量按 Agent 逐个开启，不要求全团队都有足够样本。哪个 Agent 先攒够同类任务量（建议阈值 ≥3），就先对谁产出有意义的成长快照；样本不足的 Agent SHALL 显式标注「样本不足，暂不度量」而非给出误导性趋势。

> 背景：真实数据显示当前仅数据工程师（流萤）在持续产生同类任务（4 个），其余角色仅 0–1 个。度量层天然单点先行。

#### Scenario: 样本充足的 Agent 开启度量
- **WHEN** 某 Agent 在某类型下真实工作任务数 ≥ 阈值
- **THEN** 为其产出成长快照

#### Scenario: 样本不足的 Agent 明确标注
- **WHEN** 某 Agent 真实工作任务数不足阈值
- **THEN** 返回「样本不足，暂不度量」，不给出趋势结论

### Requirement: 反事实验证探针（L3，远期）

系统 SHOULD（远期）支持用已解决的老任务作为回归集，在「带/不带某条 knowhow」两种条件下重跑并比对结果，以测量该经验的因果贡献与是否发生负迁移。此为最接近 SIP-Bench 的验证层，落实前 SHALL 先明确重跑机制与成本边界。

#### Scenario: 反事实对照重跑
- **WHEN** 对某条被高频援引的 knowhow 做因果验证
- **THEN** 在相同老任务上分别以「注入/不注入该 knowhow」重跑，比较效率与成果差异，识别正向贡献或负迁移
