# Design — Agent 自改进度量层

> 本文记录方法论与关键取舍，是后续分阶段落实的蓝本。理论依据见 `Papers/Self-Improving-Agents-Survey-核心思考.md`。

## 度量层要回答的三个问题（对齐论文 SIP-Bench）

- **Q1 能力涨没涨（纵向增益）**：同一 Agent 随时间，做**同类任务**是否更快、更少返工、更少求助。关键是「同类任务的纵向对比」，非单次分数。
- **Q2 涨的是不是真的（排除三种假象）**：
  - **过拟合 / 负迁移**：knowhow 只对特定任务有用，换场景失效甚至帮倒忙。
  - **成本转移(cost transfer)**：任务「看起来」快了，成本挪到别处（人工验收变重、token 爆、返工转移下游）。
  - **隐藏回归(hidden regression)**：学了新经验，老任务反而变差。
- **Q3 能否归因**：改进能否归因到具体原因（某条 knowhow、某次 reflect）。不能归因就无法迭代。

## 为什么选「单 Agent 为主语 + 效率代理指标」起步

- 单 Agent 贴合「让 Agent 学习成长」命题，也是论文 self-evolution 层的度量对象；团队层需先解决 10.7 的归因难题（真协同 vs 堆算力），故后置。
- 效率代理指标**全部来自现有表，零埋点**，能立刻让「变强」可见，先粗后精。

## 现有痕迹资产盘点

| 痕迹 | 现有字段 | 可算 |
|---|---|---|
| `task_runs` | status / started_at / ended_at | 单任务耗时、成功率、返工次数 |
| `activities` | 全流程状态流转 | 返工信号（reviewing→in_progress 回退次数）、人工介入频次 |
| `messages` | 每轮汇报 | 对话轮次、求助/澄清频次 |
| memory knowhow 段 | 带 `akivili:task:xx` 标记 | 沉淀条数、来源任务 |
| `solved_tasks` | 按 Agent 计数 | 累计产出（是「量」不是「变强」，仅作背景） |

缺的是把这些按 **Agent × 时间 × 任务类型** 串成纵向视角的那一层。

## L1 核心指标：knowhow 复用率

```
knowhow 复用率(单 Agent) = 被后续任务显性援引过的 knowhow 条数 / 已沉淀 knowhow 总条数
配套：
├─ 复用分布：集中在少数「金句」经验，还是普遍被用
├─ 「僵尸经验」清单：沉淀但从未被援引 —— 反向信号，指导「该怎么沉淀」
└─ 标注：本指标为「显性援引」，非因果贡献；负迁移在此层不可见（需 L3）
```

**为什么是最值钱的信号**：其他指标（耗时/返工）都是间接、可能骗人的；只有「沉淀的经验被后续任务真正用到」才直接证明「经验变成了能力」，是 Loop 闭合的硬证据。

### 「复用」的三种识别强度
- **① 呈现级（弱，排除）**：Agent 开工时记忆被注入上下文 → 「看到」几乎必然发生，看到 ≠ 用到。
- **② 引用级（中，L1 采用）**：Agent 在产出里显性援引某条经验（关键词/语义匹配 messages）。可落地，但有噪声（嘴上说用了实际没用 / 用了没明说）。
- **③ 因果级（强，L3）**：去掉该 knowhow，表现变差 —— 反事实探针，最真最贵。

L1 用 ②，**必须诚实标注天花板**：它测「显性援引」不是「因果贡献」；论文警示的负迁移（援引了却帮倒忙）在 ② 层测不出。

## 三个必须现在想清的难点

1. **任务难度不可比 → 曲线假涨假跌**：不同难度任务放同一曲线无意义。→ L1 绑定 L2 的最小分型，绕不开。
2. **耗时变短可能是成本转移**：单看耗时会误判。→ 耗时必须和「返工/求助」一起看；真变强是「又快又不返工又少求助」，单项变好存疑。
3. **样本少 → 曲线抖动**：一个 Agent 一类型可能就三五个任务。→ L1 不追求精密曲线，先做**只读诊断快照**（最近 vs 最早的粗对比），不强行拟合。

### L1 产出形态：体检快照（示意）
```
流萤(数据工程师) · 取数类任务体检
├─ 已完成同类任务：N 个（时间范围）
├─ 耗时趋势：最早均值 → 最近均值（↓ x%）
├─ 返工：早期均值 → 近期均值
├─ 求助：趋势
├─ knowhow：沉淀 M 条，其中 K 条被后续任务复用  ← 核心
└─ ⚠️ 存疑：样本偏少；某次耗时反弹的原因标注
```

## 关键设计原则：度量绝不接入 Agent 奖励（防 Gaming）

论文第10章「每个使适应成为可能的面，也能侵蚀控制」在此应验：若让 Agent 知道「被复用=好」，它可能**为刷复用率硬套旧经验**（本该用新方法却强行援引旧 knowhow）。因此：

> **本度量层的所有指标只用于人工观测与诊断，绝不作为 Agent 的直接优化目标/奖励信号。** 一旦指标变成目标，它就不再是好指标（Goodhart 定律）。

## L2 任务分型：真实数据检验后的结论

对当前库（13 个 done + 1 个 in_progress）实测后，L2 的方案与直觉不同：

**真实数据三事实**：
1. 13 个 done 里 **7 个是「自我介绍」仪式任务**（desc=0），真实工作任务仅 6 个。
2. 类型极不均衡：**仅数据工程师（流萤）攒够同类样本**（43/64/65/66 共 4 个，全是「直播日历数据」取数/补数/校验），前端 1、PO 1，其余角色 0。
3. **纯标题关键词分型会撞车**：多任务都含「直播日历」却分属取数/前端(63「年份选择组件」)/协调(62「更新数据和跨年组件」是 PO 的)。

**据此定的方案**：
- **分型 = 角色 slug（阶段A）**：每个 Agent 当前任务性质单一，角色粒度恰好 = 任务类型，零成本。实测数据工程师 4 个真实任务确实同类，角色粒度奏效。
- **两级可延展（阶段B）**：当某角色任务变杂（如 PO 既协调又碰数据/前端），在角色内再按关键词/执行痕迹分子类型。不一次定死。
- **仪式过滤 = 启发式 + run 兜底**：标题含「自我介绍/Kickoff」是强信号；但 **desc=0 不能单独作为过滤条件**——task 64 desc=0 却是真实工作（有 run）。故规则为「desc=0 且无 task_runs 才算噪声」，用「有没有真执行过」这个硬痕迹兜底。
- **单点先行原则**：度量层天然按 Agent 逐个启动。哪个 Agent 先攒够同类任务（阈值 ≥3）就先度量谁；流萤是天然 first mover。样本不足者显式标注「暂不度量」，不给误导趋势。

**关键洞察**：L2 当下真正的瓶颈**不在分型逻辑**（角色 slug 足够简单），**而在样本积累**——整个平台目前只有数据工程师一条线在持续产同类任务。这意味着度量层的价值会随「各 Agent 各自攒够任务量」而逐步铺开，而非一次覆盖全员。

## 与其他能力的关系

- `agent-reflection` 产出 knowhow（复用率的分子分母来源）。
- `agent-memory` 存储 knowhow（`memory/<slug>.md` 的 managed 段）。
- `agent-skills` 是 knowhow 的「毕业去向」：反复被复用的 knowhow 应升级为命名 Skill（论文 skill/memory 判据：变成「命名过程」→ 归 skill）。度量层可反哺「哪些 knowhow 该升级」。
- `agent-collaboration`：团队层度量的前置，需先解决协同归因。
