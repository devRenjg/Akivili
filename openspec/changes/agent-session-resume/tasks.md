## S1 — session 表 + 串行约束 + session_id 抓取（claude）

- [ ] 1.2 **同 (conversation, agent) 串行折叠（Review P0-3/P0-B；第五轮 P1-5 粒度改 conversation）**：先给 `run_queue` `ALTER ADD COLUMN conversation_id`（从 `tasks.conversation_id` 回填、新写入随 task 带入）→ 归并/取消存量同 `(conversation, agent)` 多 active 行 → 建**两组互补 partial unique index（第六轮 P1-1 NULL 兜底，SHALL NOT「不约束」）**——`UNIQUE(conversation_id,agent_slug) WHERE conversation_id IS NOT NULL AND status active` + `UNIQUE(task_id,agent_slug) WHERE conversation_id IS NULL AND status active`;新业务 run 强制带 conversation_id，无法回填的历史/系统 run 进 NULL 兜底索引或先隔离清理（应用层查重有 TOCTOU，不足以防并发 POST 各插一条）;**与 session owner 键 `(conversation_id, agent_slug)` 粒度一致**，防同 conversation 两 task 争抢 owner。折叠模型——queued 未 claim→原子 `MAX`/`COALESCE` 合并新触发消息水位到该行（不先读后写）;claimed·running→持久化 `rerun_requested`+`pending_through_message_id`+触发来源;收尾事务内据 pending 至多建一个 successor（取代现状 `collab.py:392` 的 `return None` 丢弃）。各终态折叠：failed→仍建 successor;killed→取消 pending 不建;superseded→pending 由 recovery child 继承不另建。**固化 task:conversation 1:1（第七轮 P0-3）**：迁移清历史重复后建 `UNIQUE INDEX ON tasks(conversation_id) WHERE conversation_id IS NOT NULL`，使单 pending intent 安全（否则同 conversation 多 task 折叠会压掉/错投跨 task 触发）。**NULL run 固定 full replay、不建/复用 agent_sessions（第七轮 P1-2）**：`agent_sessions` 键无 task_id、SQLite UNIQUE 对 NULL 不提供 task 级唯一性，「NULL 退化到 (task,agent) session」不可实现；NULL run 只用 task 级 active 串行、固定全量回灌、不 resume;未来确需再引 `session_scope_type+session_scope_id`，不复用 NULL conversation_id。探针 `session_grain_alignment_probe`（同 conversation 两 task 同 agent 并发触发→只一个 active、不争抢 owner）、`session_grain_null_probe`（conversation_id=NULL 的同 task/agent 并发触发→task 级兜底索引保证至多一个 active）、`task_conversation_uniqueness_probe`（两 task 复用同 conversation_id 被 DB 约束拒绝、迁移无重复）、`null_session_scope_probe`（NULL run 不复用 agent_sessions、固定 full replay、多 NULL run 不误当同一 session）
- [ ] 1.3 `claude_code.py::_parse_line` 从 `system`(init)/`result` 行提取 `session_id`,经 `ExecEvent` 新字段冒泡（现在只当「执行完成」,不丢弃）
- [ ] 1.4 session_id 生命周期 = **session owner 四阶段 CAS 协议（Review P1-3 / 第四轮 P0-4 / 第五轮 P0-2 闭环）**：三件套（pin/覆盖/COALESCE 空值保护）之上落 acquire→pin→final→retire——**(1) acquire** 新 attempt 启动时按观察到的旧 `session_version` CAS 把 `current_task_run_id` 改为本 attempt、epoch+1（`RETURNING` 拿 owner_version;首次无行用 `INSERT ... ON CONFLICT DO NOTHING` 归属本 attempt）;**acquire 五档资格谓词（第五轮 P0-2 / 第六轮 P0-1，lease 字段修正）——① `current_task_run_id IS NULL` 可 acquire；② 旧 attempt 已终态→version CAS；③ 旧 attempt claimed/preparing→仅 `run_queue.claim_lease_until<db_now` 且 PGR 回收事务已落 abandoned+退休 owner 后；④ 旧 attempt running→仅 Worker generation lease 失效+新 generation fencing+旧 attempt 回收/终态+owner 退休后；⑤ acquire 不自行凭时间过期跳过 PGR 回收事务，正常路径等回收退休 owner 后从档1接管。`task_runs` 无 lease 字段，acquire SQL SHALL NOT 读 `task_runs.lease_expires_at`;lease 词汇 = `run_queue.claim_lease_until`（claim 租约）+ `worker_state.lease_expires_at`（世代心跳租约）。active owner 不被普通 attempt 覆盖、不强抢** **(2) pin** 流中途只允许 `WHERE current_task_run_id=? AND session_version=owner_version` 更新 session_id，pin **不递增** version（epoch 只在 owner 切换/退休变）**(3) final** 成功收尾**并入统一 `finish_execution()` 同事务**（pointer+committed+successor+owner retire，final 与 retire 不分两次提交）**(4) retire** attempt 终态后清 `current_task_run_id`/推进 epoch，使迟到 pin/final CAS 失败;**终态无 successor 也 retire**;poisoned/降级清 session 与 retire 同事务。**fallback rebind（第五轮 P0-2）**：同 attempt 重建会话走 `retire old epoch→reacquire/rebind→start fresh session`，持新 token 才 pin。**lease reclaim/supersede/kill/poisoned 同步退休 owner（第五轮 P0-2）**。防新 attempt 首次 pin 不满足 WHERE 死锁 + 防迟到事件覆盖 + 防抢占 active owner

- [ ] 1.6 探针 `agent_session_build_probe`：首次执行后 `agent_sessions` 有正确 session_id、committed_msg_id、provider/backend/workdir 落库;同 (conversation, agent)（NULL 走 task 兜底）重复触发折叠不丢、不并行起两条

## S2 — claude resume + 增量回灌（committed/planned 两阶段水位，at-least-once 安全）
- [ ] 2.1 `runner.py` 执行前查 `agent_sessions`,命中且 backend/provider/workdir 一致 → 走「增量 + resume」；否则走「全量首建」
- [ ] 2.2 **2 阶段水位**：`task_runs.planned_through_msg_id` = 构建本次 prompt 那刻的 `MAX(messages.id)`（本次快照终点，落该 attempt 行）;`agent_sessions.committed_msg_id` **只在 run 成功收尾才推进到 `batch_scan_end`（本批已完整扫描并取舍的最高原始 messages.id，非最后一条 eligible），SHALL NOT 直接推进到 planned_through**（Review 第四轮 P0-2，见 2.3a）;持久字段仍是 committed+planned 两个，`batch_scan_end` 可作 attempt 内临时计算，不新增第三主水位。构建后崩溃 committed 不提前推进（否则漏）
- [ ] 2.3 增量历史 SQL：`messages WHERE conversation_id=? AND id > committed_msg_id AND id <= planned_through_msg_id AND 作者非本 agent ORDER BY id`（参数化）;崩溃/中断时 committed 未推进→续跑从同一起点重取→重复但不漏（at-least-once）;本 Agent 自产历史由 session 记忆承载不重喂
- [ ] 2.3a **🔴 海量增量连续前缀分批 + 原始扫描水位（Review P0-C / 第四轮 P0-2，禁用「保新丢旧」裁剪）**：`_clip_history` 是保留最新、丢弃较早——若套在增量上会丢中段而 committed 仍推进到 planned_through → 中段永久漏喂。改为：从 committed 之后最旧一条起按连续前缀截取本轮量，收尾 `committed_msg_id` 推进到 **`batch_scan_end`（本批已完整扫描并取舍的最高原始 messages.id，非最后一条 eligible id）**，不直接推进到 planned_through;超上限在**已完整扫描的原始区间末尾**截断、剩余未扫描尾部下一轮从新起点续喂，**禁止跳段**;「连续前缀」= 排除自产后的 eligible message 连续扫描区间（不要求原始 ID 无空洞）;单条超预算消息至少完整投递一条不卡死。**边界必须覆盖（否则空转，Review 第四轮 P0-2）**：区间全自产→committed 直接推进到 planned_through 不建 successor;eligible 尾部之后全自产→committed 跨过推进到 planned_through 不建 successor;遇上限→batch_scan_end 只到已扫描区间末尾
- [ ] 2.3b **🔴 backlog 自动续批（Review P0-4 / 第四轮 P0-2）**：run 成功且 `batch_scan_end < planned_through_msg_id`（确有未扫描原始区间）时，**同事务**（连同 committed 推进 + session pointer 更新）建 `history_backlog` successor run（`trigger=history_backlog`/`history_batch_no`/`history_batch_end`/`history_backlog_from_execution_id`），不依赖新用户 @;**`batch_scan_end` 已达 planned_through（含全自产/尾部自产跨过）时 SHALL NOT 建 successor**;successor 不计 mention-chain、受独立最大批次数/token 预算、复用同 session;全批消费完才清 backlog;达上限转人工提示不无限跑
- [ ] 2.3c **backlog 快照冻结 + pending intent 优先级（Review 第四轮 P1-5）**：backlog chain 冻结初始 `planned_through_msg_id`，chain 全程消费同一快照、不每批扩到最新 MAX（否则永远追不完）;chain 期间新用户触发只并入 pending intent、不扩当前快照;全批消费完后至多建一个普通 successor 处理 pending 新快照;用户 kill 同时取消 backlog+pending;superseded 时 backlog+pending 均由 recovery child 继承
- [ ] 2.3d **backlog 副作用安全线 = 方案 A（Review 第五轮 P0-1，用户拍板）**：① **tool-enabled 自动续批默认 feature flag 关闭**（独立上线门，不随阶段 1 默认开）;② **`history_backlog_side_effect_observe_probe` 升级为 release gate 硬门禁**——发现工具调用/平台写/普通 assistant 用户交付/mention·通知/任务自动流转即**判失败**（非只采集）;③ **上线契约二选一明确（Review 第七轮 P1-3，用户拍板）**：有可靠 safe ingestion/context-only turn→原文逐批进 session（服务端禁工具/禁交付）、投递后才推进 committed，摘要仅辅助;无可靠 safe ingestion→超单 turn 上下文直接转人工/检索，SHALL NOT 声称自动消费完成、SHALL NOT 越过未投递原消息推进 committed;摘要 SHALL NOT 既「不算消费」又「承担自动完成消费」两义并存;首批不完整上下文不得直接执行真实任务;④ **投递契约 = 原始消息至少一次完整投递（Review 第六轮 P1-7 用户拍板）**——摘要/聚合只压缩模型可见上下文、不替代原消息消费;committed 推进到 `batch_scan_end` 的前提是该区间原消息已至少完整投递过一次，不因「已生成摘要」跨过原消息推进 committed;摘要记录保留覆盖的 source message id 区间;单条或整体超模型硬上下文无法在 safe turn 内投递时停止转人工/外置检索，不无限重试、不静默丢弃后推进 committed;⑤ 仍启用轻方案则文档标注 best-effort + 记录已接受风险，不写「已解决」;无论是否启用：不计 mention-chain、不触发任务自动流转、不依赖 exactly-once。新增探针 `history_backlog_no_tool_probe`(三批含旧命令→无工具/平台写/外部副作用否则失败)、`history_backlog_no_deliverable_probe`(无普通 assistant 用户交付)、`history_backlog_incomplete_context_probe`(关键否定信息放第二批→首批不提前执行真实任务)、`history_backlog_feature_flag_probe`(flag 关闭时不自动 tool-enabled 续批)、`backlog_summary_trace_probe`(三批经摘要进单次业务 turn→committed 对应 source id 可追溯、原消息已完整投递过、硬超限转人工不静默丢弃)、`backlog_ingestion_contract_probe`(历史超一个业务 turn→只有 safe ingestion 才自动推进、无 safe ingestion 时转人工且 committed 不越过)
