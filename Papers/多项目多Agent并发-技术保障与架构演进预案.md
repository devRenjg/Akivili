# 多项目 · 多 Agent 并发规模化 — 技术保障与架构演进预案

> 目的：在「多项目、多 Agent 并行任务」规模起来之前，提前识别真实瓶颈、给出预案与演进路径。
> 本文基于对当前代码的实地核查（非泛泛而谈），全程只分析、**不做任何变更**。
> 记录日期：2026-07-09。对应能力：`agent-collaboration` / `agent-execution` / `agent-runtime-config`。

## 结论先行（TL;DR）

按「会先炸 / 影响面 / 修复成本」排序，当前架构在并发规模化时的真实风险从高到低：

| # | 风险 | 现状根因 | 触发规模 | 严重度 |
|---|------|---------|---------|--------|
| R1 | **SQLite 写锁竞争 `database is locked`** | 无 WAL、无 busy_timeout、每调用新开连接 | 几个并发写就可能撞 | 🔴 高 |
| R2 | **全局并发池只有 3 槽、无项目公平** | `MAX_CONCURRENCY=3` 单一全局池；`_claim_one` 按 `id` 全局最旧 | 多项目一起提任务即互相饿死 | 🔴 高 |
| R3 | **同一 Agent 跨项目并发写同一记忆文件** | 记忆按 slug 跨项目共享；`_claim_one` 不按 slug 去重 | 同名 Agent 在 2 项目同时跑 | 🟠 中高 |
| R4 | **每个 Agent = 一个 CLI 子进程，资源无上限治理** | 每 run 起一个 claude/codex 进程 | 并发数 × 每进程内存 | 🟠 中高 |
| R5 | **单进程单机，无水平扩展** | 全部内存态（`_running`/`_RUN_PIDS`）绑单进程 | 超单机 CPU/内存/文件句柄 | 🟠 中 |
| R6 | **记忆文件 read-modify-write 无锁** | `upsert_managed_section` 纯读改写 | 同 slug 并发 reflect | 🟡 中低 |
| R7 | **轮询式队列（1s）** | `_loop` sleep 1.0 轮询 | 任务量大时吞吐/延迟 | 🟡 低 |

**一句话**：当前架构是「单机单进程 + SQLite + 全局 3 槽池」，非常适合当前单项目验证，但**多项目并发时最先炸的是 SQLite 写锁(R1)和池公平性(R2)**，且这俩改动成本低、收益大，应作为「规模化前的第一道技术保障」。

## 逐项详解与预案

### R1 — SQLite 写锁竞争（🔴 最先炸）

**现状（实地核查）**：
- `database.py get_connection()`：每次调用 `aiosqlite.connect()` 新开连接、用完即关，只设了 `PRAGMA foreign_keys=ON`。
- **全库无 `journal_mode=WAL`、无 `busy_timeout`**（已 grep 确认，唯一命中是 collab 的 HARD_WALL 常量，无关）。
- 默认 rollback journal 模式下，SQLite 是「单写者 + 写时全库排他锁」。多个并发写事务（多 Agent 同时落 messages/activities/task_runs/run_queue）会立即 `database is locked`，且无 busy_timeout → 直接抛错而非等待。

**规模化后果**：多项目并发时，落库高频（每条流式消息、每次状态流转、每条 run_log）。并发写一多，`database is locked` 异常四起 → run 失败、状态不一致、孤儿记录激增。

**预案（低成本→高成本）**：
- **P1（首选，改动极小）**：开启 `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000`（连接建立时设）。WAL 让「读不阻塞写、写不阻塞读」，busy_timeout 让写冲突自动重试等待而非立即报错。单这一步就能扛住相当规模。
- **P2**：写操作收敛到「串行写队列」或短事务 + 重试装饰器（指数退避），进一步消除偶发锁。
- **P3（规模再上台阶）**：迁移到 Postgres（MVCC，天然并发写）。是根治，但涉及 SQL 方言、部署、迁移脚本，成本高，**非必要不做**——WAL+busy_timeout 能覆盖很长一段路。

> 关键判断：R1 用 P1 就能解决 80%，且几乎零风险。这是规模化前**性价比最高的一步**。

### R2 — 全局并发池 3 槽 + 无项目公平（🔴）

**现状**：
- `MAX_CONCURRENCY=3`，`_running` 是**单一全局集合**，不区分项目。
- `_claim_one()`：`SELECT * FROM run_queue WHERE status='queued' ORDER BY id LIMIT 1` —— 永远领**全局最老**的 run，无项目轮转。

**规模化后果**：
- 项目 A 一次性提交 20 个任务排在前面，项目 B 的任务全被饿死（要等 A 的 20 个跑完）。
- 3 槽是全局天花板：10 个项目并发也只有 3 个 Agent 在跑，其余全排队 → 体感「系统很慢/卡住」。

**预案**：
- **P1**：`MAX_CONCURRENCY` 改为可配置（Paladin/环境变量），按机器核数与内存调；先粗调到经验值。
- **P2（公平调度）**：`_claim_one` 改为「按项目轮转（round-robin）」或「每项目并发上限 + 全局上限」双层配额，避免单项目独占。
- **P3（分级队列）**：引入优先级/权重队列（如人工触发 > 自动协同；小任务 > 长跑）。
- **P4**：池大小与 R4 的资源治理联动（见下）——并发数不能超过「机器能同时承载的 CLI 进程数」。

### R3 — 同一 Agent 跨项目并发写同一记忆文件（🟠 多项目特有）

**现状（关键的多项目陷阱）**：
- 记忆按 **agent slug 跨项目共享**（`memory/<slug>.md`，设计如此，同一 Agent 无论在哪个项目记忆互通）。
- 但 `_claim_one` 只按 run_queue.id 领取，**不按 agent_slug 去重**。`enqueue_run` 的去重也只在「同 task_id + 同 slug」范围（`WHERE task_id=? AND agent_slug=?`）。
- 因此：**同一个 Agent（如数据工程师）在项目 A 和项目 B 各有一个任务时，两个 run 可以同时被领取、并发执行** → 它们会**并发读改写同一个 `memory/engineering-data-engineer.md`**（reflect 沉淀、per-run 动态写入）。

**规模化后果**：
- 记忆文件 R6 的 RMW race 被放大：两个进程同时 read→modify→write，后写覆盖先写 → **knowhow 丢失/串档**。
- 这正好会**破坏我们刚立的度量层（knowhow 复用率）**——记忆被并发写坏，血缘和条目都不可信。

**预案**：
- **P1（最简）**：加「同 slug 串行」约束——`_claim_one` 领取时跳过「该 agent_slug 已在 `_running` 里」的 run（同一 Agent 全局同一时刻只跑一个 run，跨项目也排队）。代价：同名 Agent 跨项目不能真并行，但换来记忆一致性。**推荐**，因为记忆是 Agent 的核心资产，一致性 > 并行度。
- **P2**：或者保留跨项目并行，但给记忆写加文件锁（见 R6），并把「近期动态」写入改为 append-only + 后台合并，避免全文件 RMW。
- **判断**：P1 与 P2 是「一致性优先 vs 并行度优先」的取舍，需你拍板（见文末决策点）。

### R4 — 每 Agent 一个 CLI 子进程，资源无上限治理（🟠）

**现状**：
- `executor/claude_code.py` / `codex.py`：每个 run 用 `subprocess.Popen` 起一个独立的 claude/codex CLI 进程（在线程里读流，不阻塞事件循环 —— 这点是好的）。
- 但**没有「机器级资源闸」**：并发数由 `MAX_CONCURRENCY` 间接决定，没有基于实际内存/CPU/句柄的准入控制。每个 CLI 进程可能吃数百 MB。

**规模化后果**：并发一高，N 个 CLI 进程 × 每进程内存 → 打爆机器内存/句柄，触发 OOM 或 Windows 句柄耗尽（你此前已踩过 Windows socket 句柄继承的坑）。

**预案**：
- **P1**：把「并发上限」和「机器资源」显式挂钩：设一个保守的进程数上限，文档化「每进程约 X MB，机器 Y GB → 上限 Z」。
- **P2**：加资源准入探针——起新进程前检查可用内存/负载，不足则排队等待（背压 backpressure）。
- **P3**：进程池复用 / 预热（若 CLI 支持常驻），减少反复启停开销。

### R5 — 单进程单机，无水平扩展（🟠 中）

**现状**：调度全靠**单进程内存态**（`_running` 集合、`_RUN_PIDS` 字典、`_loop` 协程）。重启即丢（已靠 `reclaim_orphan_runs` 兜底孤儿）。无法多进程/多机分担。

**规模化后果**：单机 CPU/内存/句柄是硬顶。要支撑「很多项目 × 很多 Agent」，早晚需要水平扩展，但当前内存态架构做不到多 worker。

**预案**：
- **P1（当前够用）**：先纵向扩展（更大机器）+ R1/R2/R4 优化，单机可撑很远。
- **P2（未来演进）**：把调度状态外置（run_queue 已在 DB，是好基础），`_running`/PID 也落库或放 Redis，使调度无状态化 → 可跑多 worker 进程。
- **P3（远期）**：真正的分布式执行器（worker 节点池 + 中心队列），CLI 进程分散到多节点。这是大改，**只有规模确实到那一步才做**。

### R6 — 记忆文件 RMW 无锁（🟡）

**现状**：`upsert_managed_section` / `write_memory` 是纯 `read → 正则替换 → write_text` 全量重写，无文件锁、无并发保护。

**后果**：单项目下 reflect 由「任务 done」触发、同 slug 同时刻并发概率低，暂时安全。但一旦 R3 成立（同 Agent 跨项目并发），必然踩 race。

**预案**：文件锁（`filelock` 库）或写串行化；或改 append-only 日志 + 后台 compact。与 R3 的 P2 是同一件事。

### R7 — 轮询式队列（🟡 低）

**现状**：`_loop` 用 `asyncio.sleep(1.0)` 轮询 run_queue。

**后果**：任务量大时，1s 轮询带来吞吐上限和最长 1s 的启动延迟；DB 空轮询也有微小开销。当前规模无感，规模大后是次要优化项。

**预案**：事件驱动唤醒（enqueue 时 `event.set()` 立即唤醒 loop），替代定时轮询。低优先级。

## 推荐的演进路线（分阶段，非一次到位）

> 原则：**先做低成本高收益的技术保障（不改架构），再按规模逐级演进**。绝不为想象中的规模过度设计。

**阶段 0 — 规模化前的「技术保障」（低成本，强烈建议先做）**
- R1-P1：SQLite 开 WAL + busy_timeout（几行，收益最大）
- R2-P1：并发池大小可配置
- R3-P1：同 slug 串行（保记忆一致性）——或按你决策选 P2
- R4-P1：并发上限与机器资源挂钩并文档化

**阶段 1 — 公平与背压（多项目体验）**
- R2-P2：项目轮转/双层配额调度
- R4-P2：资源准入背压
- R7：事件驱动唤醒

**阶段 2 — 解耦与水平扩展（真到单机瓶颈才做）**
- R5-P2：调度状态外置（DB/Redis），worker 无状态化
- R1-P3：评估 Postgres 迁移
- R6：记忆存储升级（锁/append-only/或迁 DB）

**阶段 3 — 分布式（远期，规模确实到了才做）**
- R5-P3：分布式执行器 + worker 节点池

## 需要你拍板的关键决策点（后续探讨）

1. **R3 取舍**：同一 Agent 跨项目并发时，选「一致性优先」（同 slug 串行、记忆绝不串档）还是「并行度优先」（加锁但允许并行）？我倾向一致性优先——记忆是 Agent 的核心资产，也是度量层的地基。
2. **数据层天花板**：是否接受「SQLite + WAL 撑到某个规模，再一次性迁 Postgres」？还是更早规划迁移？我倾向前者（WAL 能撑很远，过早迁移是浪费）。
3. **规模目标**：你心里的「多项目多 Agent」量级大概是多少（项目数 × 每项目 Agent 数 × 并发任务数）？这决定阶段 0/1 就够，还是要提前铺阶段 2。
